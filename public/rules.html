Rules

How the hierarchy processor works

Input Format

  • Edges must be in the format X→Y where X and Y are single uppercase letters (A–Z)
  • Multiple edges are separated by commas: A→B, A→C, B→D
  • Numeric nodes (1–9) are also supported in the builder but classified as invalid by the API spec

Processing Rules

1
Validation

Only edges matching [A-Z]->[A-Z] are accepted. Self-loops like A→A are rejected. Everything else goes to "Invalid Entries."

2
Deduplication

If the same edge appears multiple times (e.g., G→H, G→H), only the first is kept. Duplicates are listed separately.

3
Single Parent Rule

Each node can have at most one parent. If a node appears as a child in multiple edges, only the first-encountered parent wins.

4
Connected Components

Nodes are grouped into connected components using BFS. Each component is processed independently.

5
Cycle Detection

Each component is checked for cycles using DFS with grey/black colouring. Components with cycles show "↻ cycle" instead of a tree.

6
Tree Building

Acyclic components are rendered as nested tree structures. The root is the node with no incoming edges.

API Specification

Endpoint: POST /bfhl

Request Body:

{
  "data": ["A->B", "A->C", "B->D", "C->E"]
}

Response: Returns hierarchies, invalid entries, duplicate edges, and a summary with tree/cycle counts.
